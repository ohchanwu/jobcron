# Feature Ideas — Parked

_[Read this in English 🇬🇧](feature-ideas.md)_

논의했지만 스코프를 지키기 위해 v1에서 명시적으로 잘라낸 것들입니다. 각 항목은 다음을
답해야 합니다: 무엇인지, 왜 원하는지, 왜 v1이 아닌지, 그리고 무엇이 구현의 방아쇠가 되는지.

---

## 이력서 업로드 → 프로필 자동 추출

**무엇.** 프로필 설정 UI에 이력서(PDF / 마크다운 / 일반 텍스트)를 끌어다 놓습니다. 앱이
이를 파싱하여 스택 태그, 경력 연차, 학력, 지역 등을 추출합니다. 편집 가능한 미리보기로
추출된 프로필을 보여줍니다. 사용자가 확인하거나 편집한 뒤 저장합니다.

**왜 원하는가.** 마법 같은 첫 실행 경험. 프로필 설정의 인지 비용을 "20개 필드 폼 작성"에서
"여기에 이력서를 끌어다 놓으세요"로 낮춥니다. 신규 사용자와 데모 관람자에게 큰 "우와" 순간.

**왜 v1이 아닌가.** 완전히 새로운 문제 영역(PDF 파싱, 레이아웃 추출, 한국어 + 영어
이력서를 위한 다국어 개체명 인식(NER))을 더합니다. 첫 실행 UX 중의 파싱 실패는 치명적입니다 —
첫인상이 깨집니다. 구조화된 폼을 먼저 배포하고, 사람들이 실제로 어떻게 채우는지 본 다음,
실제 형태에 맞춰 파서를 만드는 편이 낫습니다.

**구현 방아쇠.** v1이 사용자 50명 이상을 확보했고 *동시에* 구조화된 폼 완료율이 약 70%
미만일 때(즉, 폼이 마찰이 될 때). 그러면 이력서 업로드가 명백한 해법이 됩니다.

## AI / ML 점수화 계층

**현황 (2026-06-06).** 이 아이디어의 *호스팅* 절반은 **v2.0 BYOK-AI 라인**으로 출시됐습니다 —
LLM이 각 공고를 읽고 증거 인용 기반 Stage-2 점수 델타를 반환하며, 키워드 분해 위에 더해집니다
(대체가 아니라 추가 — 아래 "빌드 트리거"가 규정한 그대로). OpenAI는 제공됐다가 **제거**됐습니다:
"v1이 아닌 이유" (1)에서 지적한 속도 제한 위험이 현실이 됐습니다 — 무료/엔트리 등급은 재평가
작업량을 감당하지 못했습니다. **이제 Anthropic이 유일한 제공자입니다.** 아직 열려 있는 방향은
**로컬 / 내장(built-in) 모델**(gguf / ONNX)입니다 — 속도 제한 없음, 파인튜닝 가능 — 현재
`task-list.md`에 설계 과제로 등록돼 있습니다.

**무엇.** 현재의 키워드·가중치 점수기를 시맨틱(semantic) 모델로 대체(또는 보강)합니다 —
프로필과 공고 간의 임베딩 + 코사인 유사도, 또는 각 공고를 읽고 짧은 근거와 함께 적합도
점수를 출력하는 LLM. 로컬(gguf / ONNX)일 수도, 호스팅(Anthropic / Together / …)일
수도 있습니다.

**왜 원하는가.** 현재 매칭은 토큰 단위로 정확히 일치합니다: "개발"은 "개발자"와 매칭되지
않고, "야근 없음"은 "야근"과 구별되지 않으며, "야근이" 같은 조사 부착 형태는 "야근"을
놓칩니다. 시맨틱 모델은 그 간극의 대부분을 메우고 키워드 매처가 잡을 수 없는 뉘앙스를
포착할 것입니다(예: "성장하고 싶은 신입 환영" ≈ "신입 친화적").

**왜 v1이 아닌가.** 세 가지 이유. (1) "단일 바이너리, 런타임에 네트워크 없음, API 키
없음"이라는 배포 명제를 깹니다 — 호스팅 LLM은 비용 처리, 키 저장, 속도 제한, 오프라인
성능 저하를 뜻합니다. (2) 현재 점수화는 의도적으로 읽기 쉽습니다 — "이건 왜 60점이지?"의
답이 세부 내역 칩에 분명히 있습니다. 시맨틱 점수는 불투명하며("모델이 그랬으니까")
차분하고 신뢰할 수 있는 제품 명제와 충돌합니다. (3) 라벨링된 정답 세트 없이는 ML 점수기가
*실제로* 더 나은지 알 수 없는데, 그 세트는 v1이 여러 출처에 걸쳐 "예/아니오, 이게 좋은
매칭이었나" 신호를 충분히 쌓을 만큼 오래 돌아가기 전까지는 존재하지 않습니다.

**구현 방아쇠.** v1이 여러 포털에 걸쳐 최소 2~3개월간 사용되었고 *동시에* 사용자가
"키워드 매처가 이걸 플래그했어야 하는데 안 했다" / "이걸 높게 매겼는데 사실 무관하다"는
명시적 사례를 가지고 있을 때. 그러면 실제 사용에서 라벨링된 평가 세트를 만들고, 그다음
모델을 그 위에 올립니다 — 절대 대체가 아니라, 항상 키워드 세부 내역과 나란히 놓이는 추가
신호로.

## 하이브리드: 구조화된 폼 + 자유 서술형 키워드 오버라이드

**무엇.** 구조화된 프로필 폼을 주 입력으로 유지하되, 그 아래에 두 개의 자유 서술형 필드를
추가합니다: "반드시 포함해야 할 키워드"(필수 포함)와 "절대 포함되면 안 되는
키워드"(dealbreakers). 실제 한국 구직 선호는 어떤 구조화된 필드에도 맞지 않는 롱테일(long-tail)
신호를 가집니다: 병특 가능, 자유 출퇴근, 노가다 거부, 적대적 완전 재택, 외국계, 스톡옵션,
외국어 사용 환경 등.

**왜 원하는가.** 구조화된 폼만으로는 모든 신호를 담을 수 없습니다. 자유 서술형 키워드는
폼 스키마를 부풀리지 않으면서 롱테일을 채웁니다. 엄격한 포함/제외 의미론은 점수화 엔진을
단순하고 예측 가능하게 유지합니다.

**왜 v1이 아닌가.** v1은 고정된 스키마 기준의 구조화된 점수화가 *충분한지*를 시험하고
있습니다. 기준선의 답을 알기 전에 자유 서술형 오버레이를 더하면 신호가 흐려집니다. 또한:
긴 형식의 공고 텍스트에 대한 제외 키워드는 신중한 매칭(토큰 경계, 부정 처리)이
필요합니다 — 그것 자체가 하나의 작은 프로젝트입니다.

**구현 방아쇠.** v1 사용자들이 구조화된 프로필이 중요한 신호를 놓친다고 보고할 때. 첫 구현은
오직 가산적(additive)이어야 합니다 — 키워드 오버라이드는 점수를 수정할 뿐, 어떤 구조화된
필드도 대체하지 않습니다.

## 추가 스크래퍼 (원티드, 프로그래머스, 사람인, 잡코리아, 회사 직접 페이지)

**무엇.** 추가 출처에 대해 `Scraper` 인터페이스를 구현하고 등록하여, 사용자가 프로필에서
어떤 것을 켤지 선택하게 합니다.

**왜 원하는가.** 사용자가 말한 문제 대비 v1의 가장 큰 약점: 점핏 하나로는
네이버/카카오/쿠팡/토스/배민/당근 커리어 페이지를 커버하지 못하고, 신입 IT 직무가 가끔
등장하는 비개발 중심 포털도 커버하지 못합니다.

**왜 v1이 아닌가.** 각 스크래퍼는 자체 DOM/API 형태, 자체 속도 제한 예절, 자체 안티봇
고려 사항을 가진 별도의 파서입니다. 원티드는 CDN 차단되어 있고(robots.txt에 403을 반환)
헤드리스 브라우저나 프록시 인프라가 필요할 가능성이 높습니다. 회사 직접 페이지는 예고 없이
바뀝니다. 유지보수 비용은 출처마다 선형으로 누적됩니다.

**구현 방아쇠.** 점핏 전용 v1이 탄탄하고 아키텍처가 버텨냈을 때. 사용자가 더 넓은 커버리지를
원하고 *동시에* 새 스크래퍼를 유지보수할 인내심이 있을 때. 릴리스당 새 출처 하나씩 추가 —
절대 "전부 스크랩" 메가 릴리스로 묶지 말 것.

## 공채 (대규모 채용) 캘린더 통합

**무엇.** 주요 한국 기업(네이버, 카카오, 삼성, LG 등)의 상반기/하반기 공채 일정을
추적합니다. 일일 브리핑에 "카카오 신입 공채 마감까지 3일" 같은 것을 표시합니다.

**왜 원하는가.** 공채 마감은 신입에게 유독 중요합니다 — 하나를 놓치면 6개월을 기다려야
합니다. 공채 캘린더를 능동적 스크래핑과 통합하는 기존 도구는 없습니다. 신입 코호트에게
명확한 차별점이 될 것입니다.

**왜 v1이 아닌가.** 선별된 공채 일정 데이터 소스가 필요합니다(수동? korecruIT에서 스크랩?
jojoldu/junior-recruit-scheduler에서 스크랩?). 선별 부담 + 통합 UI 설계 + 점핏이 대부분의
공채 공고를 담지 않는다는 사실 → 상당한 스코프 확장.

**구현 방아쇠.** v1이 배포되어 사용될 때. 커뮤니티 기여가 열릴 때 — 공채 사이클을 잘 아는
누군가가 데이터 소스를 기여하고 싶어 할 때.

## 백그라운드 스케줄러 / "설정해 두고 잊어버리는" 모드

**무엇.** 바이너리가 계속 실행되며 일정에 따라(예: 매일 아침 8시) 스크랩하고, 사용자가 앱을
여는 순간 일일 브리핑을 보여주는 선택적 데몬(daemon) 모드.

**왜 원하는가.** "차분한 아침 의식"은 브리핑이 이미 준비되어 있을 때 더욱 강력해집니다.
사용자의 아침에서 30초의 스크랩 대기를 없앱니다.

**왜 v1이 아닌가.** 다음을 더합니다: 프로세스 감시, 라이프사이클 관리, OS별 자동 시작
(launchd / systemd / Windows Service), "8시에 기기가 잠들어 있었음"에 대한 오류 처리,
스케줄러 설정 UI, 그리고 얼마나 자주 / 언제에 대한 질문들. 수동 버튼 클릭은 P2(최소 실행
가능)를 만족하며 이 모든 복잡성을 피합니다.

**구현 방아쇠.** 일일 사용자들이 명시적으로 요청할 때. v1의 수동 흐름이 최소 한 달간의
실제 사용으로 검증되었을 때.

## 알림 채널 (Slack / Discord / 이메일 / Telegram)

**무엇.** 새로운 강한 매칭 공고를 외부 채널로 푸시하여, 사용자가 알기 위해 앱을 열 필요가
없게 합니다.

**왜 원하는가.** 일부 사용자는 풀(pull)보다 푸시(push)를 선호할 것입니다. 강한 매칭이
나타나는 순간 사용자에게 도달합니다.

**왜 v1이 아닌가.** "차분한 아침 의식" 명제와 정면으로 충돌합니다. 알림은 FOMO와 조급함을
만듭니다 — 편안함의 정반대입니다. 푸시는 또한 도구라기보다 훨씬 더 제품에 가까운 형태입니다.
우리는 사용자의 웰빙이 아니라 리텐션을 위해 설계하게 됩니다. 누군가 이를 구현한다면 기본적으로
꺼져 있어야 하고 "권장되는 사용법이 아님"이라고 분명히 표시되어야 합니다.

**구현 방아쇠.** 강한 사용자 수요 *동시에* 편안함 명제를 훼손하지 않는 명확한 설계가 있을 때.

## 지원 트래커 (칸반 / 상태 보드)

**무엇.** 사용자가 지원한 공고, 현재 단계(지원완료 / 서류통과 / 1차면접 / 최종합격 / 불합격 /
자진사퇴), 면접 날짜, 후속 알림을 추적합니다.

**왜 원하는가.** 고리를 닫습니다: 도구가 일자리를 찾는 것을 돕고 *동시에* 지원을 관리하는
것도 돕습니다. JobSync와 Huntr 같은 도구가 이를 위해 존재합니다 — 명확한 수요가 있습니다.

**왜 v1이 아닌가.** 다른 제품 표면입니다(CRUD 중심, 상태 중심, 다중 페이지). v1은 "발견 +
필터링"이고, 지원 추적은 자체 UX를 가진 별도의 문제입니다. 이를 v1에 더하면 명제가
흐려집니다.

**구현 방아쇠.** v1 사용자들이 이미 다른 것(Notion, 스프레드시트)으로 지원을 추적하고 있으며
통합되기를 원한다고 보고할 때.

## 다국어 UI (영어 등)

**무엇.** 모든 UI 문자열을 번역 파일(`web/i18n/ko.json`, `web/i18n/en.json`)로
외부화합니다. 설정에서 언어 전환을 허용합니다.

**왜 원하는가.** 한국어를 쓰지 않는 사람들에게 기여의 문을 엽니다. 한국 일자리를 노리는
외국 개발자에게 유용합니다.

**왜 v1이 아닌가.** 번역 작업은 끝이 없고 v1의 대상 독자는 한국인입니다. *그러나:* v1은
1일차부터 `ko.json` 하나만 존재하더라도 문자열을 단일 파일로 외부화*해야* 합니다. 그래야
나중에 영어를 추가하는 것이 리팩터링이 아니라 커뮤니티 PR이 됩니다.

**구현 방아쇠.** 외부화는 v1 아키텍처 결정(기능이 아님)입니다. 실제 영어 번역: 기여자가
제안할 때 언제든.

## 패키징: Homebrew tap, AUR, Nix, Docker 이미지

**무엇.** `brew`, AUR, Nix에서의 일급(first-class) 설치, 그리고 그 방식을 선호하는
사용자를 위한 Docker 컨테이너.

**왜 원하는가.** 더 쉬운 설치 = 더 넓은 채택. 각 생태계에는 tarball을 절대 내려받지 않을
사용자가 있습니다.

**왜 v1이 아닌가.** 각 패키징 대상은 자체 유지보수 부담(formula 업데이트, 의존성 추적,
서명)을 가집니다. v1의 설치 방법은 "tarball 내려받아 바이너리 실행"이며 — README를 따를
의향이 있는 초기 채택자에게는 충분합니다.

**구현 방아쇠.** v1이 별 50개 이상을 받았고 *동시에* 커뮤니티의 누군가가 특정 채널을 요청할
때. 우리가 직접 유지보수하기 전에 커뮤니티가 유지보수하는 formula를 먼저 받아들일 것.

## ~~한 출처가 압도할 때의 더 넓은 경력 수준 포함~~ — 2026-05-27 배포됨(SHIPPED)

**해결.** 데모데이용으로 배포됨. `any` 버킷 사후 필터는 (1) 경력 오버라이드 파서(4년 이상
요구 시 드롭)와 (2) IT/개발 키워드 게이트(제목+직무에 개발 신호가 없으면 드롭)를
결합합니다. 200개 행 샘플링: (1)이 3개 드롭, (2)가 117개 드롭, 80개 유지(약 40% 생존),
생존자는 압도적으로 개발 직무입니다.

일반 메커니즘(출처별 행 상한, AI / 임베딩 필터)은 보류됨 — 키워드 게이트 방식이 v1에는
충분하고 데이터가 더 무거운 손잡이를 정당화하지 않았습니다. 다음의 경우 이 보류 항목을
다시 엽니다:

- 동일한 압도적 버킷 형태를 가진 두 번째 출처가 배포될 때 — 그때가 일반 해법이 그 복잡성을
  정당화하는 시점입니다.
- 사용자가 실제로 키워드 게이트가 너무 시끄럽다고 보고할 때(예: "사업개발" → 엔지니어 같은
  거짓 양성(false positive)).
- 키워드 세트가 드리프트할 때(정규식 / 한국어 목록에 없는 새 개발 전문 분야).

v1.x 결정을 다시 검토해야 할 경우를 위해 원래 프레이밍을 아래에 아카이브 목적으로 남깁니다.

**원래 프레이밍.** 일일 브리핑을 압도할 수 있는 "선호 없음" 버킷을 안전하게 포함하기 위한
출처별 정책(또는 전역 메커니즘). 구체적으로: 데모데이의 `experience_level = "any"`는 1000개
행 중 약 720개를 차지합니다 — 다수가 진정으로 신입 친화적이지만, 그것들을 모두 포함하면 한
출처가 대시보드를 범람시켜 차분한 목록 명제를 깰 수 있습니다.

**왜 원했는가.** `any`를 통째로 제외하는 것(2026-05-27 스크래퍼가 처음에 한 것)은 실제
커버리지를 잃었습니다. 그 버킷에는 사용자가 결코 보지 못하는 신입 친화적 공고가 숨어
있습니다. 같은 형태가 향후 출처에서도 나타날 수 있습니다("선호 없음" 태그를 지원하는 모든
애그리게이터(aggregator)).

**배포된 답**은 옵션 3의 더 엄격한 버전을 택했습니다: 오버라이드 파서 드롭과 작은 IT 키워드
화이트리스트를 결합. 순수 옵션 3은 98.5% 생존을 냈고(너무 관대 — 마케팅/디자인 직무로 범람),
키워드 게이트가 그것을 40%로(대부분 개발 직무) 낮췄습니다.

## 네이버페이 (네이버파이낸셜) 커리어 스크래퍼

**무엇.** `recruit.naverfincorp.com`(NAVER FINANCIAL 커리어 포털 — 네이버페이
결제·대출·보험·카드·증권·부동산 채용을 커버)을 신입 IT 공고 대상으로 스크랩합니다.

**왜 원하는가.** 어제 네이버 메인 커리어 시도가 실패한 후, 가장 기대값(EV)이 높은 회사
커리어 대상으로 처음 식별됨 — 네이버페이는 한국 신입 핀테크 엔지니어에게 최상위 목적지로
널리 여겨집니다.

**왜 v1(또는 v1.6)이 아닌가.** 2026-05-28 정찰에서 두 가지 차단 요인을 발견:

1. **활성 공고 0개.** `entTypeCdArr=0010`(신입) 필터가 0개 행을 반환합니다. 필터 없이도
   목록 페이지는 "진행 중인 공고가 없습니다"를 렌더링합니다. 비어 있는 출처에 대해
   스크래퍼를 구현하는 것은 순수한 부담입니다.
2. **네이버 메인 커리어와 동일한 사람인 스타일 포털.** 목록은 `/rcrt/list.do`에 있고,
   클릭 핸들러는 `onclick="show(annoId)"`(`/rcrt/view.do`로의 JS POST)이며,
   `copyUrl()`의 딥링크 패턴은 GET을 통한 `/rcrt/view.do?annoId=X&lang=ko`입니다. GET
   형태가 실제로 공고 본문을 렌더링하는지는 확인할 수 없었습니다(테스트할 라이브 데이터
   없음). 이는 정확히 어제 네이버 커리어를 보류한 그 아키텍처입니다.

**재정찰 방아쇠.** 3~6개월 후 `recruit.naverfincorp.com/rcrt/list.do?entTypeCdArr=0010`을
재확인. 신입 수가 0이 아니라면, 실제 브라우저 클릭으로 `/rcrt/view.do?annoId=X&lang=ko`에
대한 딥링크를 검증. 본문이 렌더링되면 스크래퍼가 가능; `/rcrt/list.do`로 리디렉트되면
(네이버 메인 커리어 실패 모드) 영구 보류. JSON API:
`GET /rcrt/loadJobList.do?entTypeCdArr=0010&firstIndex=0`이 `{result, list[], totalSize}`를
반환; robots.txt 404(RFC 9309에 따라 제한 없음).

## 토스 (Toss) 커리어 스크래퍼

**무엇.** `toss.im/career/jobs`를 신입 IT 공고 대상으로 스크랩합니다. 백엔드 ATS는
Greenhouse로 보입니다(URL 패턴이 gh_jid와 유사한 `?job_id=N`).

**왜 원하는가.** 토스는 최상위 한국 핀테크 목적지입니다; 신입 엔지니어가 주요 목표로
여깁니다.

**왜 v1(또는 v1.6)이 아닌가.** 2026-05-28 정찰에서 두 가지 복합적 이유를 발견:

1. **IT/개발 신입 공고 0개.** `/career/jobs`의 활성 공고 229개 중 신입 또는 주니어 태그를
   단 6개만 가지며 — 그 6개 **모두** 고객 서비스, 영업/MD, 프로덕트 운영, 증권 정산, 또는
   온라인검수 직무입니다. 개발/엔지니어링 신입 공고는 단 하나도 없습니다. 토스는 광범위하게
   채용 중이지만, 신입 파이프라인은 다른 곳(추천, 내부 프로그램)에 있는 것으로 보입니다.
2. **robots.txt가 직무 상세 크롤링을 권장하지 않음.** 맨 목록 `/career/jobs`는 명시적으로
   허용되지만, `Disallow: /career/jobs?*`가 필터링된 목록을 차단하고,
   `Disallow: /career/job-detail?gh_jid=5599901003`은 개별 직무 페이지가 크롤링되지 않아야
   한다는 의도를 알립니다(우리 프로젝트의 문자 그대로의 접두사 매처는
   `/career/job-detail?job_id=X`를 통과시키겠지만요). 정중한 크롤러는 문자 그대로의 규칙뿐
   아니라 그 의도를 존중해야 합니다.

**재정찰 방아쇠.** 6개월 후 신입/주니어 개발 태그 수를 재확인. IT/개발 신입 공고가 나타나고
*동시에* 토스가 직무 상세 정책을 명확히 하도록 robots.txt를 개정하면 다시 검토. 그렇지
않으면 보류 유지.

## 쿠팡 (Coupang) 커리어 스크래퍼

**무엇.** `www.coupang.jobs/kr/jobs/`(한국어 쿠팡 커리어 포털)를 신입 IT 공고 대상으로
스크랩합니다.

**왜 원하는가.** 쿠팡은 한국 최대 테크 고용주 중 하나입니다; 의미 있는 신입 IT 파이프라인이
예상됩니다.

**왜 v1(또는 v1.6)이 아닌가.** 2026-05-28 정찰에서 하드한 구조적 차단 요인을 발견:
**Cloudflare 안티봇 챌린지.** `https://www.coupang.jobs/`와
`https://www.coupang.jobs/kr/jobs/` 모두 — 평범한 `curl`과 gstack 헤드리스 Chromium 양쪽
모두에 대해 — "Attention Required! | Cloudflare" 챌린지 페이지와 함께 HTTP 403을
반환합니다. robots.txt는 허용적이므로(`User-agent: * Allow: /`) 차단 요인은 정책이 아니라
안티봇 인프라입니다.

이는 원티드(위 "추가 스크래퍼"에 이미 언급됨)와 구조적으로 동일합니다. Cloudflare를
안정적으로 우회하려면 유료 거주용 IP 프록시, 안티 탐지 기능을 갖춘 스텔스 헤드리스
브라우저(puppeteer-stealth, 핑거프린트 마스킹을 적용한 Playwright Firefox), 또는
Cloudflare의 공식 스크래핑 허용 목록 중 하나가 필요한데 — 그 어느 것도 v1의 순수 Go, CGO
없음, 외부 의존성 없음 설계에 맞지 않습니다.

**재정찰 방아쇠.** 프로젝트가 선택적(opt-in) 스텔스 헤드리스 스크래핑 경로(예: Chromium
서브프로세스를 활성화하는 `JOBCRON_USE_HEADLESS=1` 플래그)를 채택하면, 쿠팡과 원티드가
함께 가능해집니다. 그전까지는 하드 차단.

것입니다.

**구현 방아쇠.** 프로젝트가 안정적인 유지보수 체계를 갖추고 *동시에* 예산이 있을 때.
